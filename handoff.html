WO-to-Proposal Matching (Tier 2)

The Link Proposal enhancement inside the Control Center Qualify Ticket (QT) task — surfaced inline, in the current DMG design system.

Prototype: qt_proposal_match_all_states.html — open and use the dark bar at the top to switch states.
Requirement: Confluence — Requirements: WO-to-Proposal Matching Agent (Tier 2), FUL space, page 4363583489.
Surfaces: dmg.cc-tasks.ui / dmg.fulfilment.ui (new QT task type Link Proposal); intelligence from IRIS; link/approve/attribution via ticket-service + proposal-service.

What this is

When a customer approves a DMG proposal, it often arrives as a brand-new work order with no reference to the original proposal — so we lose attribution and can't measure proposal→approval conversion. The Tier 2 agent runs during qualification, detects when a WO likely came from a submitted proposal, and surfaces it to the NAT inline in the Qualify Ticket panel (no new page or modal).

Match signals (strongest → weakest)

  1. NTE proximity — WO NTE vs proposal customer price (strongest)
  2. Service line match — same trade
  3. Scope similarity — WO description vs proposal title/scope
  4. Timing — proposal recency

Shown as compact chips (✓ green = strong, ≈ amber = partial). Every match screen shows the same set: Same store · Same trade · NTE proximity · Scope match.

Behavior rules (all match screens)

  1. Qualification actions are gated. While a match decision is pending, the QT buttons (Update Ticket and Continue, Update NTE, Save & Next, Qualification Done) are greyed/disabled. They enable only when the NAT rejects the match and goes manual.
  2. Link / Confirm → auto-qualify, after a confirmation step stating the outcome: "We'll mark PRP-XXXXXX approved and auto-qualify this ticket. A job will be created and posted to the preferred provider — [Provider]. If they decline or aren't available, it moves to the job board." No preferred provider → posts to the job board. On confirm: proposal → Approved, attribution written, ticket qualified, task status flips In Progress → Done.
  3. Dismiss → manual qualification. Rejecting enables the normal QT form; nothing is linked.
  4. Customer NTE is NOT copied from the proposal. Proposal is source of truth for scope/service/type (pre-filled + locked), but Customer NTE keeps the work order's own value and stays editable — the customer may send a WO with a different/updated price they intend to keep.
  5. Reason capture only on Auto-link → Unlink, as free text (up to 150 words) — unlinking overrides an automated decision and feeds model training. Single/Multiple dismiss is one-click.
  6. View proposal link on every proposal, showing the originating ticket (proposal was submitted on a different WO), e.g. View proposal ↗ · submitted on TKT-260401-3391.

Copy

  • Titles: Auto-linked proposal · Did this come from a proposal? · Did this come from one of these proposals?
  • Say line skeleton: "This work order looks like it came from a proposal your team already sent…" + type-specific action clause.
  • Confidence chip: PLEASE REVIEW (auto) · LIKELY MATCH (single/multiple).
  • Outcomes block (2 lines, every screen): ✓ primary (link → qualify + job) · ↩ secondary (dismiss/unlink → manual).

Design system notes

Built to the current Control Center look: DMG navy top bar (#123A5E) + orange accent (#EE7127), Poppins headings + Inter body, orange step accent bars, underline inputs, navy radios/buttons. Maps to dmgKit (Material UI): match card ≈ Card, chips ≈ Chip, buttons ≈ Button, selectable list ≈ RadioCard, confirm boxes ≈ inline Alert.

⚠️ Colors/fonts are approximated from a product screenshot; exact dmgKit theme tokens are not exposed via MCP. Swap the CSS :root variables for the real dmgKit token export for pixel fidelity.

Implementation notes

  • IRIS returns proposal_match: action (auto_link | confirm | multi_confirm | log_only | no_match), candidates (id, confidence, rationale, signals), and target proposal id for auto_link/confirm.
  • ticket-service: add WO NTE to the IRIS payload; read proposal_match; auto_link → execute link + emit event + write attribution; confirm/multi_confirm → create the pre-populated Link Proposal task; log_only/no_match → continue.
  • proposal-service: on link-confirmed event → proposal Submitted → Approved + write attribution; signal preferred provider (from ERQ) to assignment.
  • Approval & attribution fire on the NAT's confirm, not on auto-link — so Unlink is a cheap, reversible rollback (proposal returns to its prior ticket).
